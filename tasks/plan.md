# Implementation Plan: Zabbix MAX Alert Bot

## Overview

План описывает дальнейшее развитие проекта в формате `planning-and-task-breakdown`. Цель — вести задачи маленькими проверяемыми шагами, чтобы человек и AI-агент одинаково понимали порядок работ, критерии приемки и границы текущего этапа.

Текущий этап остается прежним: только доставка уведомлений Zabbix -> МАХ через Zabbix Media type `Webhook`.

## Architecture Decisions

- Основной рабочий артефакт: `src/zabbix-media-type/max-webhook.js`.
- Telegram-канал не заменяется и не ломается. МАХ добавляется как дополнительный канал доставки.
- Архитектурные решения фиксируются только в `docs/decisions/`.
- Документация ведется по `docs/documentation-policy.md`.
- Внешний `agent-skills` используется ссылкой и локальной установкой, без submodule.
- Задачи ведутся в `tasks/plan.md` и `tasks/todo.md`; `.agents/` остается рабочим контекстом агента, а не местом хранения задач.

- [ ] Task 1: Проверить MAX webhook на тестовом получателе.
- [ ] Task 2: Зафиксировать результат Problem/Recovery.

### Phase 3: Проверки качества

- [ ] Task 5: Добавить минимальную проверку форматирования сообщения.

### Checkpoint: Готовность к пилоту

- [ ] Выполнен `bash scripts/verify-repo.sh`.

- [ ] Нет незадокументированных изменений поведения webhook.

### Phase 4: Возможное развитие после пилота

- [ ] Task 7: Оценить необходимость отдельного bot-service.
- [ ] Task 8: Описать варианты повторной отправки и журнала доставки.

### Checkpoint: Решение о развитии

- [ ] Есть подтвержденные результаты пилота.
- [ ] Если нужен отдельный сервис, создан ADR в `docs/decisions/`.
- [ ] Текущий webhook-вариант не усложнен без необходимости.

## Risks and Mitigations

| Risk | Impact | Mitigation |
|---|---|---|
| MAX API или формат ответа отличается от ожиданий | Medium | Проверять через тестовый чат и фиксировать фактический ответ в документации |
| В документацию попадут чувствительные значения | High | Использовать только обезличенные примеры и проверять `scripts/verify-repo.sh` |
| AI-агент начнет расширять проект за пределы этапа | High | Перед задачей проверять `docs/decisions/README.md`, `AGENTS.md` и этот план |
| Задачи станут слишком крупными | Medium | Делить задачи до размера S/M и не выполнять L/XL без новой декомпозиции |
| Поведение webhook изменится без обновления документации | Medium | Любое изменение `max-webhook.js` сверять с `docs/zabbix-media-type.md` |

## Open Questions

- Какой тестовый получатель МАХ использовать для первичной проверки.
- Нужно ли хранить отдельный обезличенный пример ответа MAX API.
- Нужен ли отдельный документ для эксплуатационной поддержки после пилота.

## Parallelization Opportunities

Безопасно выполнять параллельно:

- подготовку документации по получателю;
- подготовку export/import инструкции;
- review документации.

Последовательно выполнять:

- проверку webhook;
- фиксацию Problem/Recovery;
- изменение логики `max-webhook.js`.

- [ ] Указан estimated scope.
